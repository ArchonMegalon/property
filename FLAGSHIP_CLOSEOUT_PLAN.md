# Standalone PropertyQuarry Flagship Closeout Plan

## Purpose

Define the remaining proof required before the standalone PropertyQuarry product can make a flagship-grade launch claim.

This repository contains inherited Executive Assistant governance names and product-canon references. Those names do not broaden the proof target. The current browser evidence intentionally skips legacy Executive Assistant office-loop tests, so it cannot establish Executive Assistant core eligibility.

## Current verdict

PropertyQuarry browser release eligibility is fail-closed until the current generated receipts are green and bound to the current PropertyQuarry gate seed. A green browser receipt proves only the exact source-backed and real-browser nodes named by that seed:

- `tests/test_propertyquarry_workspace_redesign.py` proves the standalone search/workspace surfaces and failed-run recovery state.
- `tests/e2e/test_propertyquarry_greenfield_browser.py` proves the ranked-result/research journey and mobile usability.
- `.codex-studio/published/EA_BROWSER_WORKFLOW_PROOF.generated.json` must identify product `propertyquarry`, proof target `propertyquarry`, and those exact current file/case nodes.
- `.codex-design/product/EA_FLAGSHIP_RELEASE_GATE.generated.json` must agree with the current PropertyQuarry seed and truth plane.

These receipts do not prove `/app/today`, briefing, inbox, approval, commitment, handoff, people-memory, activation, memo, draft, or any other legacy Executive Assistant workflow. They also do not replace the full PropertyQuarry release gates, release-bound disaster-recovery evidence, deployment provenance, live authenticated checks, provider coverage, or operator approval.

## Flagship blockers

### PQ-FG-001 — browser proof is current and exact

Status: release-gated.

A previously published pass cannot survive a failing current run. Both proof lanes must execute all seed-required nodes without failures, errors, skips, xfails, or unexpected passes. The published product and proof target must match the current seed.

### PQ-FG-002 — search and research surfaces are customer-safe

Status: covered only to the extent of the named PropertyQuarry browser nodes.

The proof covers the standalone PropertyQuarry workspace, ranked results, research packet opening, mobile usability, and actionable failed-run state. Broader product behavior remains subject to its own contracts and live gates.

### PQ-FG-003 — recovery and launch operations are proven

Status: release-gated.

Launch authority requires a release-bound encrypted Postgres backup, an immutable provider object version, provider-verified retrieval of that exact version, a disposable restore, schema/integrity/data evidence, and RPO/RTO proof. Local backup files are not off-host recovery evidence.

### PQ-FG-004 — runtime and provider claims match live evidence

Status: release-gated.

Static/source preflight is not a live launch verdict. Authenticated live smokes, release provenance, the current provider matrix, and every required product gate must pass for the candidate actually being promoted.

## Remaining out-of-scope claims

The PropertyQuarry flagship receipts do not promote any of the following without their own passing release-bound evidence:

- legacy Executive Assistant office workflows;
- optional LTD, commercial-operations, or publication lanes;
- memorial/avatar/video products;
- Chummer rules, desktop/UI, hub/web, or mobile surfaces;
- optional media/provider accounts that have no current account-use receipt.

## Release execution order

1. Freeze the candidate identity and require a clean, reproducible worktree/image provenance chain.
2. Run source-level PropertyQuarry contracts without treating them as live proof.
3. Produce release-bound backup and exact-version off-host restore evidence.
4. Run the explicit full PropertyQuarry release gate against the candidate and deployed runtime.
5. Review blockers, receipts, provider limitations, and rollback readiness before promotion.

## Flagship release rule

PropertyQuarry is not flagship-grade merely because `MILESTONE.json`, an inherited EA receipt, CI, or a polished page is green.

It is eligible for a standalone PropertyQuarry flagship claim only when:

- the current seed, browser proof, and generated PropertyQuarry receipt agree on the exact proof nodes;
- the full PropertyQuarry source, security, recovery, live, and provenance gates pass for the same candidate;
- required provider limitations are represented honestly rather than hidden by an older receipt;
- launch and rollback evidence are reviewed by the operator.

Any claim about legacy Executive Assistant core eligibility requires a separate seed and passing evidence that actually executes those workflows.
